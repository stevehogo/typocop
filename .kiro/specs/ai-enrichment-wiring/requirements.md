# Requirements Document

## Introduction

The indexing pipeline already contains an `AIClient` interface and `inferClusterName` function in `src/indexer/clustering/enrichment.ts` that support AI-powered cluster naming via an LLM. However, no concrete `AIClient` implementation is ever constructed, and `PipelineConfig.aiClient` is always `undefined` when the pipeline is invoked from the CLI executor. As a result, all clusters always receive heuristic labels instead of AI-generated names.

This feature wires up the AI enrichment path end-to-end: reading configuration from environment variables, constructing a real `AIClient` that calls the OpenAI chat completions API, passing it through `PipelineConfig`, and ensuring graceful degradation when the API key is absent or enrichment is explicitly disabled.

## Glossary

- **AIClient**: The interface defined in `src/indexer/clustering/enrichment.ts` with a single method `generateText(prompt: string): Promise<string>`.
- **Executor**: The `src/cli/executor.ts` module that constructs `PipelineConfig` and calls `runIndexingPipeline`.
- **OpenAIEnrichmentClient**: The concrete implementation of `AIClient` that calls the OpenAI chat completions API.
- **PipelineConfig**: The configuration object accepted by `runIndexingPipeline`, which already contains an optional `aiClient` field.
- **Heuristic label**: The cluster name produced by keyword-matching logic in `classifyCluster`, used as a fallback when no `AIClient` is available.
- **AI_ENRICHMENT_ENABLED**: An environment variable flag (`"true"` / `"false"`) that explicitly enables or disables AI enrichment independent of key presence.
- **OPENAI_API_KEY**: The environment variable holding the OpenAI secret key, already used by the embeddings pipeline.

## Requirements

### Requirement 1: Environment-based enrichment configuration

**User Story:** As a developer running the indexing CLI, I want AI enrichment to activate automatically when my OpenAI key is present and the feature flag is set, so that I do not need to modify source code to enable richer cluster names.

#### Acceptance Criteria

1. WHEN `OPENAI_API_KEY` is set and `AI_ENRICHMENT_ENABLED` is `"true"`, THE Executor SHALL construct an `OpenAIEnrichmentClient` and assign it to `PipelineConfig.aiClient`.
2. WHEN `OPENAI_API_KEY` is absent, THE Executor SHALL set `PipelineConfig.aiClient` to `undefined` and proceed without AI enrichment.
3. WHEN `AI_ENRICHMENT_ENABLED` is `"false"` or unset, THE Executor SHALL set `PipelineConfig.aiClient` to `undefined` regardless of whether `OPENAI_API_KEY` is present.
4. THE Executor SHALL read `OPENAI_API_KEY` and `AI_ENRICHMENT_ENABLED` from `process.env` without requiring any other configuration source.
5. WHEN `AI_ENRICHMENT_ENABLED` is set to any value other than `"true"`, THE Executor SHALL treat it as disabled.

---

### Requirement 2: OpenAI chat completions client implementation

**User Story:** As a developer, I want a concrete `AIClient` implementation backed by the OpenAI chat completions API, so that cluster names are generated by an LLM rather than keyword heuristics.

#### Acceptance Criteria

1. THE `OpenAIEnrichmentClient` SHALL implement the `AIClient` interface defined in `src/indexer/clustering/enrichment.ts`.
2. WHEN `generateText` is called, THE `OpenAIEnrichmentClient` SHALL send the prompt to the OpenAI chat completions endpoint using model `gpt-4o-mini`.
3. THE `OpenAIEnrichmentClient` SHALL be constructed with an API key string and SHALL use that key for all requests.
4. THE `OpenAIEnrichmentClient` SHALL return the text content of the first choice from the chat completions response.
5. THE `OpenAIEnrichmentClient` SHALL reside in a new file `src/indexer/clustering/openai-client.ts` and export the class as a named export.

---

### Requirement 3: Graceful degradation on AI failure

**User Story:** As a developer, I want the indexing pipeline to complete successfully even when the OpenAI API is unavailable or returns an error, so that a network issue or quota exhaustion does not break the entire index run.

#### Acceptance Criteria

1. WHEN `generateText` throws an error, THE `inferClusterName` function SHALL catch the error and return the heuristic label unchanged.
2. WHEN the OpenAI API returns an HTTP error response, THE `OpenAIEnrichmentClient` SHALL throw a typed `AIClientError` so that `inferClusterName` can catch and handle it.
3. IF an AI enrichment call fails for one cluster, THEN THE Pipeline SHALL continue enriching the remaining clusters using the same `AIClient` instance.
4. THE `inferClusterName` function SHALL NOT propagate any error thrown by `AIClient.generateText` to its caller.

---

### Requirement 4: Privacy — only symbol names and kinds are transmitted

**User Story:** As a user of the indexing tool, I want confidence that no source code is sent to OpenAI during cluster enrichment, so that my codebase remains private.

#### Acceptance Criteria

1. THE `inferClusterName` function SHALL call `verifyEnrichmentPrompt` on the constructed prompt before invoking `AIClient.generateText`.
2. WHEN `verifyEnrichmentPrompt` throws, THE `inferClusterName` function SHALL return the heuristic label without calling `AIClient.generateText`.
3. THE prompt sent to `AIClient.generateText` SHALL contain only the heuristic label and a list of symbol names paired with their kinds, with a maximum of 20 symbols.
4. THE prompt sent to `AIClient.generateText` SHALL NOT contain file paths, source code, signatures, or documentation strings.

---

### Requirement 5: Enrichment forwarded through the pipeline

**User Story:** As a developer, I want the `aiClient` constructed in the executor to reach the cluster enrichment step inside the pipeline, so that the wiring is complete end-to-end.

#### Acceptance Criteria

1. WHEN `PipelineConfig.aiClient` is defined, THE `runIndexingPipeline` function SHALL pass it to `clusterSymbols`.
2. WHEN `PipelineConfig.aiClient` is `undefined`, THE `runIndexingPipeline` function SHALL pass `undefined` to `clusterSymbols`, preserving existing heuristic-only behaviour.
3. THE `enrichCluster` function in `src/enrichment/index.ts` SHALL forward the `AIClient` from its `_config` parameter to `enrichClusterImpl` instead of ignoring it.
4. WHEN `PipelineConfig.aiClient` is defined and enrichment succeeds, THE resulting `Cluster.name` SHALL differ from the heuristic label for at least one cluster in a codebase with identifiable symbol patterns.

---

### Requirement 6: Verbose logging for enrichment activity

**User Story:** As a developer running the CLI with `--verbose`, I want to see whether AI enrichment is active and how many clusters were enriched, so that I can confirm the feature is working.

#### Acceptance Criteria

1. WHEN `verbose` is `true` and `PipelineConfig.aiClient` is defined, THE Executor SHALL log a message indicating that AI enrichment is enabled before starting the pipeline.
2. WHEN `verbose` is `true` and `PipelineConfig.aiClient` is `undefined`, THE Executor SHALL log a message indicating that AI enrichment is disabled and heuristic labels will be used.
3. WHEN `verbose` is `true`, THE Pipeline SHALL log the count of clusters that received an AI-generated name after Phase 4 completes.
